# AI 本地路线智能规划测试用例

## TC01：基础自然语言输入生成路线

### 测试目标

验证用户输入游玩目标后，系统可以生成完整路线。

### 用户画像

用户 A：约会偏好型。

### 输入

```text
今晚想在上海静安约会，人均 200，想安静一点。
```

### 预期结果

系统应输出 2-3 条路线，每条路线包含：

- 路线名称
- 多个 POI
- 总预算
- 总耗时
- 步行距离
- 推荐理由
- 风险提示

### 验收标准

- 路线中至少包含 2 个 POI。
- POI 城市为上海。
- 区域优先匹配静安。
- 人均预算接近 200。
- 推荐理由中出现安静、约会等关键词。

---

## TC02：路线自动串联多个 POI

### 测试目标

验证系统不是只推荐单店，而是能串联多个目的地。

### 输入

```text
周末下午想在上海静安逛逛，顺便吃饭和喝咖啡。
```

### 预期结果

系统生成类似路线：

```text
静安寺附近展览 → 青木日料 → 花园咖啡馆
```

### 验收标准

- 至少生成 3 个 POI 组成的路线。
- POI 类型有差异，比如展览、餐厅、咖啡。
- 路线顺序合理，不应明显绕路。
- 显示每个 POI 的停留时间或到达时间。

---

## TC03：预算约束测试

### 测试目标

验证系统能根据预算筛选路线。

### 用户画像

用户 A：约会偏好型。

### 输入

```text
今晚上海静安约会，人均 100 以内，不想排队。
```

### 预期结果

- 系统优先推荐低人均餐厅或轻食/咖啡组合。
- 不应推荐明显超预算的高端餐厅。
- 如果路线略超预算，需要明确提示。

### 验收标准

- 总预算小于等于 100，或明确展示“略超预算”。
- 如果略超预算，需要给出解释，例如：

```text
这条路线预算更低，但餐厅氛围会比高端路线弱一些。
```

---

## TC04：时间约束测试

### 测试目标

验证系统能考虑营业时间和用户出行时间。

### 输入

```text
今晚 9 点后想在上海静安吃饭，再找个地方坐坐。
```

### 预期结果

- 推荐的 POI 必须在 21:00 后仍营业。
- 不应推荐 21:00 前关门的店。
- 如果可选地点少，需要说明原因。

### 验收标准

- 路线中每个 POI 都满足：用户到达时间早于 POI 关闭时间。
- 页面应展示营业状态，例如：

```text
营业中 / 营业至 22:00 / 深夜可去
```

---

## TC05：排队约束测试

### 测试目标

验证“不想排队”可以影响推荐。

### 输入

```text
今晚上海静安约会，不想排队，安静一点。
```

### 预期结果

- 优先推荐 queue_minutes 较低的 POI。
- 避免推荐 risk_tags 包含“排队久”的商户。
- 如果推荐热门店，需要提示排队风险。

### 验收标准

路线推荐理由中应出现：

```text
排队较短
等位风险低
避开热门排队店
```

---

## TC06：UGC 评价语料参与推荐

### 测试目标

验证系统能利用用户评价语料，而不是只看评分。

### 前置数据

A 店：

```text
评分 4.8
UGC：人很多，排队久，环境吵
```

B 店：

```text
评分 4.6
UGC：环境安静，适合约会，排队短
```

### 输入

```text
想找适合约会、安静一点的餐厅。
```

### 预期结果

系统应优先推荐 B 店，而不是只因为 A 店评分更高就推荐 A 店。

### 验收标准

推荐理由中明确出现：

```text
来自评价标签：安静、适合约会、排队短
```

## TC07：结合历史偏好生成个性化方案

### 测试目标

验证同样输入，不同用户得到不同路线。

### 输入

```text
周末下午帮我安排一条上海路线。
```

### 用户 A 预期结果

用户 A：约会偏好型。

```text
安静咖啡馆 → 日料餐厅 → 适合散步/拍照的地点
```

推荐理由包含：

```text
匹配你的历史偏好：安静、日料、适合约会
```

### 用户 C 预期结果

用户 C：探店内容型。

```text
网红甜品店 → 买手店 → 出片咖啡馆
```

推荐理由包含：

```text
匹配你的历史偏好：网红、出片、新店
```

### 验收标准

- 同一句输入下，用户 A 和用户 C 的路线不同。
- 推荐理由明确展示不同历史偏好。
- 至少有一条路线标记为“最符合你的偏好”。

---

## TC08：生成多条差异化方案

### 测试目标

验证系统不是生成三条类似路线，而是多维度差异化。

### 输入

```text
今晚上海静安约会，人均 200，想安静一点。
```

### 预期结果

系统输出至少三类方案：

```text
方案 A：综合最优
平衡评分、预算、距离和偏好。

方案 B：少走路
优先缩短步行距离和总耗时。

方案 C：偏好优先
优先匹配用户历史偏好的安静、日料、少排队。
```

### 验收标准

三条路线必须有明显差异：

- 路线目标不同。
- POI 组合不同。
- 推荐理由不同。
- 对比指标不同。

不能只是换个名字但路线内容几乎一样。

## TC09：动态调整 - 更便宜

### 测试目标

验证用户可以基于已有路线进行预算约束调整。

### 初始输入

```text
今晚上海静安约会，人均 200，想安静一点。
```

### 继续输入

```text
能不能更便宜一点？
```

### 预期结果

- 系统保留原始场景和区域。
- 降低路线总预算。
- 替换部分高价 POI。
- 给出调整说明。

### 验收标准

- 调整后路线总预算低于原路线。
- 调整后路线仍在上海静安附近。
- 调整后路线仍尽量保留约会/安静偏好。

---

## TC10：动态调整 - 少走路

### 测试目标

验证系统能根据空间约束调整路线。

### 初始输入

```text
周末下午上海静安逛街吃饭喝咖啡。
```

### 继续输入

```text
少走路一点。
```

### 预期结果

- 新路线 POI 更集中。
- 总步行距离下降。
- 推荐理由说明“地点更集中”。

### 验收标准

- 调整后 walking_distance_meters 小于调整前 walking_distance_meters。
- 页面展示：

```text
已为你减少步行距离
```

---

## TC11：动态调整 - 不想排队

### 测试目标

验证系统能基于排队风险重新生成路线。

### 初始输入

```text
今晚想吃热门一点的餐厅。
```

### 继续输入

```text
但我不想排队。
```

### 预期结果

- 替换掉 queue_minutes 高的餐厅。
- 或提示热门与不排队存在冲突。
- 给出低排队备选路线。

### 验收标准

- 调整后路线的 queue_minutes 明显降低。
- 或系统提示：

```text
热门和不排队存在冲突，你可以优先选择“热门”或“少排队”。
```

---

## TC12：冲突条件识别

### 测试目标

验证系统能识别用户条件冲突。

### 输入

```text
想找人均 80 以内、网红、高评分、不排队、适合约会的餐厅。
```

### 预期结果

系统不应直接硬生成不靠谱方案，而是提示条件冲突。

预期提示：

```text
这些条件可能存在冲突：低预算、网红、高评分、不排队。
你更想优先保证哪一个？
```

选项：

```text
预算优先
不排队优先
网红出片优先
高评分优先
```

### 验收标准

- 系统识别至少 2 个冲突条件。
- 系统给用户优先级选择。
- 用户选择后，系统重新生成路线。

## TC13：信息不足时主动追问

### 测试目标

验证系统不会在信息严重不足时乱推荐。

### 输入

```text
周末想出去玩。
```

### 预期结果

系统应该追问，而不是直接生成路线。

预期追问：

```text
你想在哪个城市？
更偏向吃饭聊天、拍照逛街，还是亲子/看展？
预算大概是多少？
```

### 验收标准

- 至少追问城市或场景。
- 问题数量不要太多，建议 1-2 个核心问题。
- 用户回答后可以继续生成路线。

---

## TC14：信息基本足够时少打断

### 测试目标

验证系统不会过度追问。

### 输入

```text
今晚在上海找个适合约会的地方。
```

### 预期结果

系统可以默认预算和区域，直接生成路线，同时说明假设。

预期文案：

```text
我先按人均 150-250、适合约会、上海热门商圈来推荐。
```

### 验收标准

- 不应强制用户填写所有信息。
- 可以基于合理默认值生成路线。
- 明确展示系统假设。

## TC15：历史偏好解释可见

### 测试目标

验证个性化不是隐藏在算法里，而是能被用户看到。

### 用户画像

用户 A：喜欢安静、日料、少排队。

### 输入

```text
帮我安排今晚上海静安路线。
```

### 预期结果

路线卡片里必须出现：

```text
匹配你的历史偏好：安静、日料、少排队
```

或：

```text
因为你过去经常选择“不排队”和“安静餐厅”，所以这条路线优先降低等位风险。
```

### 验收标准

每条路线至少有一个解释字段：

```text
preference_match_reason
```

## TC16：POI 数据缺失时的降级策略

### 测试目标

验证系统在没有足够 POI 时不会崩溃。

### 输入

```text
今晚想在上海一个很冷门的区域找高评分日料，不排队，人均 100。
```

### 前置条件

该区域 POI 很少，或没有符合条件的 POI。

### 预期结果

系统应放宽条件，并说明原因。

预期文案：

```text
当前区域符合条件的地点较少，我为你放宽到附近商圈，并优先保留“不排队”和“预算”条件。
```

### 验收标准

- 系统不报错。
- 页面不空白。
- 系统能生成降级路线，或给出可选追问。

---

## TC17：收藏/历史影响后续推荐

### 测试目标

验证收藏和历史行为可以沉淀为偏好。

### 操作步骤

1. 用户收藏一条“安静日料 + 咖啡”路线。
2. 用户下次输入：

```text
周末帮我安排一下。
```

### 预期结果

系统优先推荐类似风格路线。

### 验收标准

推荐理由包含：

```text
参考你之前收藏的安静日料/咖啡路线
```

或者偏好标签中增强：

```text
安静
日料
咖啡
```

## TC18：多维度路线对比

### 测试目标

验证路线方案可以被用户比较。

### 输入

```text
今晚上海静安约会，人均 200。
```

### 预期结果

路线对比维度包括：

- 总预算
- 总耗时
- 步行距离
- 排队风险
- 评分
- 历史偏好匹配度
- 推荐理由

### 验收标准

用户能看出：

- 哪条更便宜。
- 哪条更省时。
- 哪条更符合偏好。
- 哪条风险更高。
