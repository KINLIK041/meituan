  二、值得优化的部分（从产品化角度）

  1. 前端 Mock 数据与后端 POI 脱节

  后端 MockDataService 有 400 个 POI，前端 mock-data/ 也有 400 个 POI——它们是两套独立数据。这导致：
  - 后端生成的路线里 POI 可能没有对应的前端图片
  - Tags 体系不统一
  - 维护两套数据容易出错

  建议：前端 mock 数据应来自后端 /api/route/health 或一个 /api/pois 端点，而不是独立维护。

  2. constraint_match 展示缺失

  需求 §10 的路线卡片规范里明确有：

  "constraint_match": {
    "budget": "略超预算",
    "queue": "符合",
    "open_time": "符合",
    "distance": "适中"
  }

  当前 RouteOption 完全不展示这些。评委看到路线卡片时会问："我怎么知道这条路线符合了我说的预算？怎么知道不排队？"

  3. 偏好分数在前端不透明

  PreferenceScorer 在后端计算评分，但前端：
  - 只展示匹配了哪些 tags（"安静、日料、少排队"）
  - 不展示分数（如"偏好匹配度：92 分"）
  - 不展示为什么这条路线比另一条更匹配

  4. Demo 流程的完整性

  需求 §11 设计了 3 个 Demo。检查当前是否能跑通：

  ┌────────────────────────────────┬───────────────────────────────────────────┐
  │              Demo              │                当前可行性                 │
  ├────────────────────────────────┼───────────────────────────────────────────┤
  │ Demo 1：同需求不同用户不同结果 │ ✅ 技术可行（UserPickerSheet + 偏好驱动） │
  ├────────────────────────────────┼───────────────────────────────────────────┤
  │ Demo 2：同用户不同约束不同结果 │ ✅ 技术可行（quick-adjust chips）         │
  ├────────────────────────────────┼───────────────────────────────────────────┤
  │ Demo 3：同需求多方案对比       │ ⚠️ 可生成但无法对比（缺对比视图）         │
  └────────────────────────────────┴───────────────────────────────────────────┘

  5. 长尾场景覆盖不足

  测试用例里的边界场景：
  - "9 点后找地方坐坐"（深夜营业）— 后端有 TimeWindowChecker 但需要验证是否正确处理
  - "冷门区域" — 需要验证降级逻辑是否真正工作

  三、架构对比：1 主 Agent + 工具 vs 5 个 Agent

  建议书的设计（1 主 Agent + 工具）

  Route Concierge Agent（路线管家）
    ├─ parse_user_intent
    ├─ get_user_profile
    ├─ search_pois
    ├─ get_travel_times
    ├─ generate_routes
    ├─ check_constraints
    ├─ score_by_preference
    ├─ rank_routes
    └─ explain_routes

  核心理念：Agent 负责思考和调度，工具负责执行和计算。LLM 处理 NLU、冲突判断、解释生成；查询、计算、校验交给确定性工具。

  当前架构（5 个 Agent）

  RoutePlannerOrchestrator（编排器）
    ├─ ConversationAgent   → LLM 解析意图
    ├─ DiscoveryAgent      → POI 搜索
    ├─ PlanningAgent       → 路线生成
    ├─ ConstraintAgent     → 约束校验
    └─ ExplanationAgent    → 推荐解释

  对比分析

  ┌──────────────┬──────────────────────────────────────────┬────────────────────────────────────────────────┐
  │     维度     │            1 主 Agent + 工具             │                   5 个 Agent                   │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 概念简洁度   │ ⭐⭐⭐⭐⭐ 一个 Agent 概念，对外讲得清楚 │ ⭐⭐⭐ 需要解释编排逻辑                        │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 评委印象     │ "AI-native 架构"，加分                   │ "传统微服务编排"，不加分                       │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 上下文完整性 │ ⭐⭐⭐⭐⭐ 所有信息在一个 LLM 上下文中   │ ⭐⭐⭐ 信息分散，靠代码传递                    │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 动态决策能力 │ ⭐⭐⭐⭐⭐ LLM 自主决定调哪些工具        │ ⭐⭐ 固定流水线                                │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 并行性能     │ ⭐⭐ 工具通常串行调用                    │ ⭐⭐⭐⭐ 已有并行优化（speculative discovery） │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 调试可测性   │ ⭐⭐⭐ 出问题时需分析 LLM 决策链         │ ⭐⭐⭐⭐ 每个 Agent 独立可测                   │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ Token 成本   │ ⭐⭐⭐ 单次调用上下文大                  │ ⭐⭐⭐⭐ 每次调用上下文小                      │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 扩展性       │ ⭐⭐⭐⭐⭐ 加工具很简单                  │ ⭐⭐⭐ 改流水线需要改编排器                    │
  ├──────────────┼──────────────────────────────────────────┼────────────────────────────────────────────────┤
  │ 异常处理     │ ⭐⭐ 依赖 LLM 判断错误情况               │ ⭐⭐⭐⭐ 每个阶段有明确的错误处理              │
  └──────────────┴──────────────────────────────────────────┴────────────────────────────────────────────────┘

  我的判断：建议书的设计更好，但不应完全推倒重来

  理由：

  1. 这个项目是比赛 Demo，不是生产系统。 评委看的是概念展示，不是压测数据。1 主 Agent + 工具的概念更容易讲出"AI
  决策"的故事。
  2. 工具调用模式更灵活。 假如用户说"我想换个便宜的"，主 Agent 可以只调用 search_pois +
  check_constraints，而不是走整个流水线。当前 5 Agent 流水线对每次请求都全量跑。
  3. 符合行业趋势。 OpenAI Function Calling、Anthropic Tool Use、LangChain Agent
  都是这个范式。评委如果熟悉这个领域，第一反应就是"这是标准做法"。
  4. 但不要全部扔掉。 当前的 5 个 Agent 已经是很好的工具实现了。建议的迁移策略：

  当前状态                          建议状态
  ──────────────────────────────────────────────────
  ConversationAgent (LLM)    →    吸收进主 Agent（LLM 自己理解）
  DiscoveryAgent (规则)      →    保留为 search_pois 工具
  PlanningAgent (规则)       →    保留为 generate_routes 工具
  ConstraintAgent (规则)     →    保留为 check_constraints 工具
  ExplanationAgent (LLM)     →    吸收进主 Agent（LLM 自己解释）
  PreferenceScorer (规则)    →    保留为 score_by_preference 工具

  本质变化：不是删除代码，而是把原来固定顺序的编排器（Orchestrator）替换为一个 LLM 驱动的 Agent
  Loop，让它动态决定调用哪些工具。

  // 当前（固定流水线）
  result = conversationAgent.process(query)
         → discoveryAgent.discover(intent)
         → planningAgent.plan(discovery, intent)
         → constraintAgent.analyze(routes)
         → explanationAgent.explain(routes)

  // 建议（Agent Loop）
  while (not done):
      action = llm.decide(context, available_tools)
      result = execute_tool(action)
      context += result

  迁移代价评估

  ┌───────────────────────────────┬───────────────────────────────────┐
  │           改动范围            │              工作量               │
  ├───────────────────────────────┼───────────────────────────────────┤
  │ 新建 AgentLoop / ToolRegistry │ 中                                │
  ├───────────────────────────────┼───────────────────────────────────┤
  │ 将现有 Agent 包装为 Tool 接口 │ 小（每个 Agent 已有清晰输入输出） │
  ├───────────────────────────────┼───────────────────────────────────┤
  │ 调整前端展示（无感知）        │ 零                                │
  ├───────────────────────────────┼───────────────────────────────────┤
  │ 调 Prompt                     │ 中（需要写好工具描述和决策逻辑）  │
  ├───────────────────────────────┼───────────────────────────────────┤
  │ 回归测试                      │ 中                                │
  └───────────────────────────────┴───────────────────────────────────┘

  总估：2-3 天可以完成架构迁移，同时保留所有现有功能。

  五、一句话总结

  ▎ 项目骨架很完整（5 Agent + 偏好系统都有了），但还差两个"评委必看"的功能：UGC 语料推荐（TC06）和多维度路线对比（TC18）。架构上，建议书推荐的"1 主 Agent + 工具"在比赛场景下比当前
  ▎ 5 Agent 更好——更好讲、更灵活、更"AI-native"，而且不需要推倒重来，把现有 Agent 包装成工具即可。
