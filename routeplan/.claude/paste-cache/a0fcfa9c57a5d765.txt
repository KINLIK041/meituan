# 项目：AI 本地智能路线规划系统（美团黑客松赛道五）

## 1. 项目目标
实现一个基于多 Agent 协同的智能路线规划后端服务。用户用自然语言描述出行/游玩意图，系统自动串联合适的 POI（餐饮、娱乐、文化等），生成满足时间、预算、偏好、排队等多维约束的最优路线方案，并支持自然语言实时动态调整路线。

## 2. 核心技术栈要求
- **语言版本**：Java 24（请充分利用虚拟线程、记录模式、switch 模式匹配等新特性）
- **框架**：Spring Boot 3.4+，使用 WebFlux 以充分利用虚拟线程
- **LLM 集成**：LangChain4j 1.0+，支持 OpenAI 兼容接口（可后续切换美团内部模型）
- **图算法**：JGraphT 1.5+，用于 POI 图构建和路径搜索
- **约束求解**：可选 Google OR-Tools 或自研轻量约束满足引擎
- **构建工具**：Maven，Java 24 编译配置
- **数据存储**：内存 + 本地 JSON 文件，预留 MySQL/PostgreSQL 接口
- **测试**：JUnit 5，Mockito

## 3. 系统架构
遵循 **层次化多 Agent 自治系统**，包含以下 Agent 和核心组件：

### 3.1 Agent 层
- **ConversationAgent**：处理多轮对话，维护上下文状态（历史选择、当前会话偏好），输出标准化的用户意图结构体。
- **DiscoveryAgent**：接收意图结构体，从 POI 数据源检索候选集（使用 `DataService`），应用硬约束过滤。
- **PlanningAgent**：接收候选集，构建 POI 连通图，调用图搜索求解器生成 2-3 个差异化路线方案。
- **ConstraintAgent**：负责将自然语言约束结构化，对生成方案进行硬/软约束验证，冲突时触发消解与重规划。
- **ExplanationAgent**：为每个方案生成自然语言推荐理由和差异说明。

Agent 间通过消息/事件驱动（可用 Spring Cloud Stream 或简单事件总线）。采用 **ReAct 模式**（推理→行动→观察循环），每个 Agent 在需要时调用工具（搜索 POI、计算路径、检查营业时间等）。

### 3.2 核心服务
- **RoutePlannerOrchestrator**：主控服务，串接 Agent 调用，支持任务 DAG 并行执行（例如同时检索不同区域 POI）。
- **IntentParser**：基于 LLM 从自然语言提取结构化意图（景点/餐饮偏好、时间窗、预算、人数、排队容忍度等）。
- **ConstraintEngine**：管理硬约束（营业时间、可达性）和软约束（预算、评分、排队时长），支持优先级分层和动态松弛。
- **GraphSearchSolver**：应用 VRPTW 思想，在 POI 图上寻找帕累托最优路径（最小化路程 + 最大化 POI 质量 + 满足约束）。
- **StateManager**：管理会话状态，保存规划中间结果（Snapshots），支持增量修改（如“换掉这家餐厅”）时局部重优化。

## 4. 数据源双轨设计
实现 `DataService` 接口，提供两个实现：

1. **DianpingApiDataService**：调用大众点评开放平台 API（需配置 apiKey/secret），获取实时 POI、UGC、排队数据。使用 WebClient 异步调用。
2. **MockDataService**：加载本地预置的 JSON 数据集（北京、上海各 200+ POI，含分类、坐标、评分、人均、营业时间、标签），用于 API 不可用时的降级演示。

通过 Spring Profile（`dianping` / `mock`）切换，确保演示零中断。

## 5. 混合路线求解算法（LLM + 图搜索）
采用学术界验证的 **LLM-as-Parser + 多步图搜索** 混合架构：

1. **LLM 解析阶段**：`IntentParser` 将用户输入转为结构化 JSON（包含：区域、餐饮类型、预算上限、开始时间、结束时间、路线类型“步行/开车”、额外偏好）。
2. **图构建阶段**：`GraphBuilder` 根据候选 POI 生成带权有向图，节点=POI，边=交通时间（调用地图 API 或估算），节点权重=POI 质量分。
3. **路径搜索**：使用分支定界或 A* 变体，考虑时间窗（营业时间窗）和顺序约束，产生 3 个帕累托方案分别优化“体验最优”“最省时”“最低花费”。
4. **LLM 解释阶段**：`ExplanationAgent` 将路径转为自然语言，附带每个 POI 的推荐理由（基于 UGC 高频词）。

## 6. 约束系统升级
- **硬约束**：营业时间必须在活动窗口内（含用餐时长）、POI 间距必须在交通预算内。不满足直接剪枝。
- **软约束**：预算、评分、口味偏好、排队容忍度。使用加权打分，冲突时按优先级自动放松（预算→评分→排队）。
- **冲突消解**：如果严格约束无解，自动逐级放松并提示用户，生成“最接近方案”并标注差异。

## 7. 反馈闭环与动态调整
- 会话中任意调整约束（如“换家不排队的火锅”），系统保留先前规划状态，仅对受影响的 POI 重新局部搜索。
- 支持自然语言指令：“前面两家不变，最后一家换成附近评分更高的咖啡馆”，系统切回 `PlanningAgent` 局部重规划。
