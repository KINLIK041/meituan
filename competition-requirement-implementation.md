# 大赛要求落地方案：差异化约束 + 用户历史偏好 + 差异化路线生成

## 1. 我们要满足的大赛要求

大赛要求可以拆成三句话：

1. 系统要能理解用户这一次提出的条件约束。
2. 系统要能结合用户历史偏好。
3. 系统要能生成多条差异化方案，而不是三条差不多的路线。

我们这个项目要表达的产品能力是：

> 用户只需要说一句模糊或半完整的需求，系统会结合当前约束和历史偏好，生成多条目标不同、可比较、可调整的本地生活路线方案。

比如用户输入：

```text
今晚想在上海静安约会，人均 200，不想排队，想安静一点。
```

系统应该输出类似：

```text
方案 A：综合最优
平衡评分、预算、距离、排队和约会氛围。

方案 B：少走路
优先缩短步行距离，适合不想折腾。

方案 C：最符合你的历史偏好
优先匹配用户过去喜欢的“安静、日料、少排队、拍照好看”。
```

重点不是只推荐店，而是完成一次“从模糊需求到可执行路线”的决策闭环。

## 2. 产品架构建议

从产品化角度，建议采用：

```text
一个主 Agent + 多个工具
```

不要真的做很多独立 Agent 互相调用。更合理的方式是：

```text
Route Concierge Agent 路线管家 Agent
  ├─ parse_user_intent       理解用户当前需求
  ├─ get_user_profile        获取用户历史偏好
  ├─ search_pois             搜索候选商户/地点
  ├─ get_travel_times        计算地点之间距离和时间
  ├─ generate_routes         生成候选路线
  ├─ check_constraints       检查预算、时间、排队、营业状态
  ├─ score_by_preference     计算历史偏好匹配分
  ├─ rank_routes             排序并选出差异化方案
  └─ explain_routes          生成推荐理由
```

主 Agent 负责：

- 理解用户说什么
- 判断信息够不够
- 决定要不要追问
- 决定调用哪些工具
- 组织最终结果

工具负责：

- 查数据
- 算距离
- 算预算
- 检查营业时间
- 打分排序
- 保存用户偏好

产品表达上可以这样说：

> Agent 负责思考和调度，工具负责执行和计算。大模型主要处理自然语言理解、冲突判断和解释生成，查询、计算、校验这些确定性任务交给工具，保证稳定性、速度和成本可控。

## 7. 当前约束和历史偏好的关系

路线生成要同时看两类信息。

### 当前约束

当前约束来自用户这一次输入。

例如：

```text
城市：上海
区域：静安
时间：今晚
预算：人均 200
场景：约会
要求：不想排队、安静
```

当前约束一般是硬约束或强约束。

比如：

- 城市必须匹配
- 营业时间必须满足
- 预算尽量不超过
- 用户说不想排队，就要降低高排队商户权重

### 历史偏好

历史偏好来自用户过去的点击、收藏、调整、跳过、对话表达。

例如：

```text
喜欢安静
喜欢日料
喜欢拍照好看
不喜欢排队
不喜欢走路远
预算通常在 150-250
```

历史偏好一般是软约束。

它不一定必须 100% 满足，但会影响排序。

可以这样表达：

```text
当前约束决定“能不能选”，
历史偏好决定“优先选哪个”。
```

## 8. 偏好匹配分如何计算

我们需要一个简单可解释的 preference score。

### 单个 POI 的偏好分

假设用户画像：

```json
{
  "preference_tags": {
    "安静": 0.90,
    "少排队": 0.85,
    "日料": 0.72
  },
  "avoid_tags": {
    "排队久": 0.90,
    "太吵": 0.80
  }
}
```

某个 POI：

```json
{
  "name": "青木日料",
  "tags": ["日料", "安静", "适合约会", "少排队"],
  "risk_tags": []
}
```

那么它匹配：

```text
安静 0.90
少排队 0.85
日料 0.72
```

这个 POI 的偏好分较高。

另一个 POI：

```json
{
  "name": "网红甜品店",
  "tags": ["网红", "出片", "热门"],
  "risk_tags": ["排队久", "太吵"]
}
```

对于约会偏好型用户，它会被扣分：

```text
排队久 -0.90
太吵 -0.80
```

### 简单打分公式

可以先用简单公式：

```text
poi_preference_score =
  命中的 preference_tags 权重之和
  -
  命中的 avoid_tags 权重之和
```

例如：

```text
青木日料 = 0.90 + 0.85 + 0.72 = 2.47
网红甜品店 = -0.90 - 0.80 = -1.70
```

### 路线偏好分

一条路线由多个 POI 组成。

```text
route_preference_score =
  路线中所有 POI 偏好分的平均值
```

也可以加入路线级因素：

```text
route_score =
  评分分
  + 偏好匹配分
  + 距离分
  + 预算分
  + 排队分
```

建议先用可解释版本：

```text
综合最优分 =
  30% 评分
  25% 偏好匹配
  20% 预算匹配
  15% 距离/时间
  10% 排队风险
```

不用一开始追求算法复杂，重点是能讲清楚。

## 9. 差异化方案怎么生成

不能只生成三条相似路线。

建议固定生成三类路线：

### 方案 A：综合最优

目标：

```text
平衡评分、预算、距离、排队、用户偏好。
```

排序目标：

```text
综合分最高。
```

展示文案：

```text
这条路线在评分、预算、距离和你的偏好之间比较均衡，适合作为默认推荐。
```

### 方案 B：效率优先 / 少走路

目标：

```text
优先减少步行距离和总耗时。
```

排序目标：

```text
总距离短、地点集中、交通方便。
```

展示文案：

```text
这条路线地点更集中，步行距离更短，适合不想折腾或时间比较紧的情况。
```

### 方案 C：偏好优先

目标：

```text
最符合用户历史偏好。
```

排序目标：

```text
preference score 最高。
```

展示文案：

```text
这条路线优先匹配你历史偏好的“安静、日料、少排队、拍照好看”。
```

## 14. 如何对外讲这个项目

可以用这段话：

> 这个项目不是一个普通聊天机器人，而是一个本地生活 AI 决策助手。用户输入模糊需求后，系统会先识别当前约束，比如城市、区域、时间、预算和偏好；再读取用户历史偏好，比如喜欢安静、少排队、日料、少走路等；然后调用商户搜索、地图距离、路线生成和约束检查工具，生成综合最优、效率优先和偏好优先三类差异化路线方案。这样用户既能获得个性化结果，也能在不同目标之间做选择。

更短版本：

> 我们用“当前约束 + 历史偏好”驱动路线生成。当前约束保证方案可执行，历史偏好保证方案个性化，差异化目标函数保证输出不是三条相似路线，而是可比较的多方案。

2. 给 POI 补充 tags / risk_tags。
3. 路线卡片展示“匹配历史偏好”。
4. 固定输出三类路线：综合最优、少走路、偏好优先。

### P1：让逻辑更可信

1. 增加 preference score。
2. 增加约束满足状态。
3. 增加不同用户同输入不同输出。
4. 增加用户切换器。

### P2：更接近真实产品

1. 从收藏、点击、调整行为里动态更新 preference tags。
2. 接入真实地图距离。
3. 接入真实商户数据。
4. 增加长期记忆和个性化推荐解释。

---

## 16. 最终验收标准

做完后，Demo 需要能证明三件事：

### 1. 能满足当前约束

用户说预算、区域、时间、不排队，路线里要体现这些限制被检查过。

### 2. 能结合历史偏好

路线卡片要展示：

```text
匹配你的历史偏好：安静、日料、少排队
```

或者：

```text
因为你过去经常选择少走路路线，所以这条方案优先减少步行距离。
```

### 3. 能生成差异化方案

至少三条路线要有明显不同定位：

```text
综合最优
少走路
偏好优先
```

评委看到后要能马上理解：

```text
这不是随机生成三条路线，
而是根据不同目标函数生成的可比较方案。
```

---

## 17. 一句话总结

我们要做的不是“推荐几家店”，而是：

> 基于用户当前约束和历史偏好，生成多条差异化、可解释、可调整的本地生活路线方案。
